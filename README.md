# Учёт движения денежных средств (ДДС)

Веб-сервис для учёта денежных операций: создание, просмотр с фильтрацией и удаление записей о движении денежных средств, а также управление справочниками (статусы, типы, категории, подкатегории) с соблюдением логических зависимостей между ними.

Проект реализован на Django + Django REST Framework. Бизнес-часть (записи) построена по принципам **Domain-Driven Design** со слоистой архитектурой; справочники — как простой CRUD на дженериках DRF.

## Возможности

- **Создание записи ДДС** с полями: дата, статус, тип, категория, подкатегория, сумма, комментарий.
- **Просмотр списка записей** с фильтрацией по периоду дат, статусу, типу, категории и подкатегории.
- **Удаление записи**.
- **Управление справочниками**: добавление, редактирование и удаление статусов, типов, категорий и подкатегорий.
- **Логические зависимости**: категория привязана к типу, подкатегория — к категории. При создании записи проверяется, что выбранная категория относится к указанному типу.
- **Валидация данных** на стороне сервера: обязательность полей, положительность суммы, существование связанных объектов.

## Архитектура

Модуль записей построен по слоистой DDD-архитектуре. Зависимости направлены внутрь: доменный слой не знает ни о Django, ни о DRF.

```
presentation  ──>  logic (services)  ──>  domain (entities)
     │                                          ▲
     │                                          │
     └──────────>  infrastructure (repositories, mappers, ORM)
```

```
First_IT_Company/DDS/
├── models.py                      # Django ORM-модели (деталь хранения)
├── migrations/
└── app/
    ├── domains/transaction/       # доменные сущности и value-объекты (чистый Python)
    │   ├── entity.py
    │   └── value_objects.py
    ├── logic/services/            # сценарии и бизнес-правила (без Django)
    │   ├── transaction_service.py
    │   └── exceptions.py
    ├── infrastructure/            # реализация хранения
    │   ├── repositories/          # репозитории + перевод доменных ошибок
    │   └── mappers/               # ORM-модель ↔ доменная сущность
    └── presentation/              # HTTP-слой (DRF)
        ├── views/                 # вьюхи (records — APIView, справочники — generics)
        ├── serializers/           # входные/выходные сериализаторы
        └── urls/
```

Каждый слой решает свою задачу: домен описывает сущности и правила, сервис их оркестрирует, репозиторий переводит между ORM и доменом через мапперы, а слой представления отвечает только за HTTP и перевод доменных ошибок в коды ответов.

## Быстрый старт

### Локально через uv

Требуется установленный [uv](https://docs.astral.sh/uv/).

```bash
# 1. Установить зависимости (uv сам создаст виртуальное окружение по uv.lock)
uv sync

# 2. Применить миграции — создаст файл db.sqlite3 и таблицы
uv run python manage.py migrate

# 3. (опционально) создать суперпользователя для доступа в админку
uv run python manage.py createsuperuser

# 4. Запустить сервер разработки
uv run python manage.py runserver
```

Сервис будет доступен по адресу `http://127.0.0.1:8000/`, админка — `http://127.0.0.1:8000/admin/`.

### Через Docker

```bash
# 1. Собрать образ
docker build -t dds-app .

# 2. Применить миграции
docker run --rm dds-app uv run python manage.py migrate

# 3. Запустить сервер
docker run -p 8000:8000 dds-app
```

Сервис будет доступен по адресу `http://127.0.0.1:8000/`.

> Чтобы база сохранялась между перезапусками, смонтируйте том:
> `docker run -p 8000:8000 -v dds_data:/app/data dds-app`
> (при этом путь к базе в `settings.py` должен указывать в `/app/data`).

## Заполнение тестовыми данными

Перед созданием записей нужно завести справочники — через админку (`/admin/`) или через API справочников. Пример минимального набора:

- **Типы**: «Списание», «Пополнение»
- **Статусы**: «Бизнес», «Личное», «Налог»
- **Категории**: «Маркетинг» (тип «Списание»), «Инфраструктура» (тип «Списание»), «Выручка» (тип «Пополнение»)
- **Подкатегории**: «Avito» (Маркетинг), «VPS» (Инфраструктура), «Продажи» (Выручка)

После этого можно создавать записи ДДС (см. примеры ниже).

## API

Базовый префикс: `/api/dds/`.

### Записи ДДС

| Метод | URL | Описание |
|-------|-----|----------|
| `GET` | `/api/dds/transactions/` | Список записей с фильтрацией |
| `POST` | `/api/dds/transactions/` | Создание записи |
| `DELETE` | `/api/dds/transactions/` | Удаление записи (id в теле запроса) |

**Создание записи** — `POST /api/dds/transactions/`

```json
{
    "status_id": 1,
    "type_id": 1,
    "sub_category_id": 1,
    "amount": "3000.00",
    "comment": "Реклама Avito"
}
```

Поле `comment` необязательно. Сумма должна быть больше нуля. `type_id` обязан соответствовать типу категории, к которой относится выбранная подкатегория.

**Ответ** (`200 OK`):

```json
{ "status": true }
```

**Список записей** — `GET /api/dds/transactions/`

Возвращает массив записей:

```json
[
    {
        "id": 1,
        "created_at": "2025-01-10",
        "amount": "3000.00",
        "comment": "Реклама Avito",
        "status": "Бизнес",
        "type": "Списание",
        "category": "Маркетинг",
        "sub_category": "Avito"
    }
]
```

**Удаление записи** — `DELETE /api/dds/transactions/`

```json
{ "id": 1 }
```

### Справочники

Для каждого справочника доступен полный CRUD. `<id>` — идентификатор записи.

| Справочник | Список / создание | Чтение / изменение / удаление |
|------------|-------------------|-------------------------------|
| Статусы | `GET, POST /api/dds/references/statuses/` | `GET, PUT, PATCH, DELETE /api/dds/references/statuses/<id>/` |
| Типы | `GET, POST /api/dds/references/types/` | `.../types/<id>/` |
| Категории | `GET, POST /api/dds/references/categories/` | `.../categories/<id>/` |
| Подкатегории | `GET, POST /api/dds/references/sub-categories/` | `.../sub-categories/<id>/` |

При создании категории указывается `type` (id типа), при создании подкатегории — `main_category` (id категории). Существование связанных объектов проверяется автоматически.

## Примеры фильтрации

Фильтры передаются как query-параметры и комбинируются по логике «И».

| Запрос | Что вернёт |
|--------|-----------|
| `GET /api/dds/transactions/?date_from=2025-02-01&date_to=2025-02-28` | Записи за февраль |
| `GET /api/dds/transactions/?status_id=1` | Записи со статусом id=1 |
| `GET /api/dds/transactions/?type_id=2` | Записи с типом id=2 |
| `GET /api/dds/transactions/?category_id=2` | Записи категории id=2 |
| `GET /api/dds/transactions/?sub_category_id=4` | Записи подкатегории id=4 |
| `GET /api/dds/transactions/?type_id=1&status_id=1` | Пересечение: тип 1 и статус 1 |

## Логические зависимости и валидация

- **Категория → тип**: каждая категория привязана к типу.
- **Подкатегория → категория**: каждая подкатегория привязана к категории.
- При создании записи доменный слой проверяет, что выбранная подкатегория ведёт к указанному типу. При несоответствии возвращается `400` с описанием ошибки.
- Несуществующие связанные объекты (подкатегория, статус) приводят к осмысленным ответам `400` / `404`, а не к ошибке сервера.
- Сумма обязательна и должна быть положительной; обязательны также тип, статус и подкатегория.

Ошибки обрабатываются по слоям: инфраструктура поднимает свои исключения, сервис переводит их в доменные, слой представления — в HTTP-коды.

## Проектные решения

- **Записи ДДС реализованы по DDD** (домен, сервисы, репозитории, мапперы) — чтобы продемонстрировать слоистую архитектуру, изоляцию домена от фреймворка и явную проверку бизнес-правил.
- **Справочники реализованы на дженериках DRF** — это тривиальный CRUD без бизнес-логики, для него полная DDD-обвязка избыточна. Такой выбор сделан осознанно: инструмент подобран под сложность задачи.
- **База данных — SQLite** для простоты локального запуска, как допускает ТЗ. Архитектура за счёт репозиториев не привязана к конкретной СУБД и допускает замену.
